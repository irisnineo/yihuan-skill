# 异环资源规划

根据玩家库存、档案偏好、版本资料和本次目标生成资源规划报告。

## 能力分层

本技能按五层维护。新增资料或规则时，先判断所属层级，再写入对应位置。

| 层级 | 内容 | 维护位置 |
| --- | --- | --- |
| 游戏概念层 | 资源、卡池、抽卡、商店、付费、好感度、都市玩法等长期稳定机制 | `references/fixed-rules.md`、`references/shop-exchange.md` |
| 版本事实层 | 当前版本的卡池时间、资源收入、活动、礼包、版本校验和版本专属活动档位 | `references/versions/v<版本>/main.md`、`references/versions/v<版本>/progress.md` |
| 玩家档案层 | 玩家目标优先级、安全垫、付费习惯、武器态度、好感度和都市偏好 | `profiles/<档案名>.md` |
| 分析流程层 | 从输入到报告的标准计算步骤、目标匹配和输出结构 | 本文件 |
| 问题分支层 | 缺数据、目标不明确、卡池时间不足、礼包未明确购买、版本中途输入等处理方式 | 本文件的“分支处理” |

## 资料读取

1. 读取游戏概念层：`references/fixed-rules.md`。
2. 读取版本事实层：`references/versions/v<版本>/main.md`；存在 `progress.md` 时同时读取，用于卡池时间、日期拆分和整版本汇总。
3. 只在涉及商店、体力、活力、材料性价比或方斯补差时读取 `references/shop-exchange.md`。
4. 读取玩家档案层：用户指定 `profiles/<档案名>.md` 时使用指定档案；未指定时使用 `profiles/nineo.md`。用户本次输入优先于档案。
5. 生成 HTML 报告时使用 `templates/resource-report.html`。

用户输入可按 `当前资源`、`氪金预算`、`目标` 分段。`氪金预算` 中明确写“计划买 / 会买 / 预算买 / 买”的礼包视为计入礼包列；写“可考虑 / 可能买 / 备选”的礼包只作为备选说明。用户写 `买 6r 方斯礼包` 或 `买 6 元方斯礼包` 且未指定数量时，按当版本 2 个角色单抽方斯礼包计入；报告展示统一写作 `购买 6 元方斯礼包 x2`，不加额外确认状态。

## 玩家档案维护

只有用户明确发出 `给{name}建档` 指令时，才建立新玩家档案。建档时复制 `profiles/profile-template.md`，保存为 `profiles/<name>.md`。

建档流程：

1. 从指令中提取 `{name}`，生成文件 `profiles/<name>.md`。
2. 复制 `profiles/profile-template.md` 的结构。
3. 将用户本次提供的长期偏好写入新档案；未提供的项目保留模板占位。

未出现 `给{name}建档` 指令时，不创建新档案。当前库存、当前版本目标、临时预算、单次礼包购买、某个版本临时跳过或追加角色、一次性买房或载具目标，都保留为本次输入或报告备注。

## 资料维护流程

新增资料时按以下顺序处理：

1. 判定层级：长期机制进游戏概念层、版本日历进版本事实层、玩家偏好进玩家档案层。
2. 确保 `references/versions/v<版本>/` 存在；缺少 `progress.md` 时复制 `references/version-progress-template.md`。
3. 从版本日历整理 `卡池信息`：`卡池名称` 填卡池横幅标题，`卡池内容` 填图中明确展示的角色名或武器名。
4. 整理每日、周期、活动、一次性和礼包资源，写清资源类型、数量、可获得日期、档位和领取方式；礼包资源同时记录 `礼包类型` 和游戏内显示名。
5. 对齐整版本汇总；存在差异时写入校验状态或待确认项。

## 标准分析流程

1. 确认版本、档案、当前库存、氪金预算和目标。库存必须来自用户输入或档案。
2. 确认资源计算方式：
   - 默认使用 `进行中拆分`，查询日期为当前日期；用户指定日期时用用户日期。
   - 用户写 `整版本预估`、`版本预估` 或 `版本前` 时，使用整版本汇总。
   - 用户写“版本 X + 当前资源”时，默认理解为规划版本 X 的期初库存；若输入表达为“当前剩余资源 / 已领取后的余量 / 版本进行中余量”，按版本中途状态处理。
3. 按玩家档案选择收入档位、目标优先级、安全垫和付费习惯；用户本次输入覆盖档案。
4. 匹配目标与卡池。角色目标匹配 `限定棋盘`，武器目标匹配 `弧盘研摹`；抽卡目标按具体角色或武器拆成独立目标，不合并成总数。
5. 计算可用资源：
   - 进行中拆分：`当前库存 + 截至查询日已开放资源 + 查询日后未开放资源 + 计入礼包`。
   - 整版本预估：`当前库存 + 整版本收入 + 计入礼包`。
6. 计算目标消耗：角色、武器、活动、好感度、都市消费和计入商店兑换。好感度消耗优先读取玩家档案 `好感度` 表，按 `价格 x 次数` 计算每日方斯消耗；免费云朵道具和免费约会按固定规则计入，表中空白或 `-` 视为未记录。
7. 按卡池结束时间检查角色和武器目标。卡池结束后的资源不用于该卡池目标。
8. 根据环石、方斯、安全垫和目标优先级给出推荐路线与备选路线。
9. 生成报告，先给结论，再给关键数字、表格和分支说明。

## 卡池与目标匹配

- `限定棋盘` 对应限定角色目标，消耗 `限定骰子` 和环石补差。
- `弧盘研摹` 对应武器目标，消耗 `三重钥匙` 和环石补差。
- `上半`、`下半`按同类型卡池时间顺序匹配；例如上半角色匹配第 1 个 `限定棋盘`，上半武器匹配同期第 1 个 `弧盘研摹`，下半角色匹配第 2 个 `限定棋盘`。
- 武器目标指定角色或写 `专武` 时，匹配与该角色同期的 `弧盘研摹`；报告项目名统一写作 `武器 <角色名>专武`，不展示具体武器名。
- 抽卡目标输出一目标一行：抽几个角色/武器目标就写几行，例如 `角色 安魂曲`、`武器 安魂曲专武`、`角色 卡厄斯`。
- 多个卡池目标按结束时间顺序滚动计算；同一结束时间先列角色、后列武器；已分配给前序目标的抽卡道具和环石不再重复计入后序目标。
- 每个抽卡目标按自身卡池结束时间单独分析资源；报告备注按“报告生成流程”的目标消耗规则输出。

## 报告生成流程

- 默认生成 HTML 到 `reports/name-yyyyMMdd.html`，标题为 `档案名 <版本> 版本资源分析`。
- 整版本预估报告副标题写作 `版本开始前库存预估：...`。
- 顶部元信息中，`版本` 只写版本号，例如 `版本：1.1`；整版本预估时，`查询日期` 写 `整版本预估`。
- 顶部元信息中，`档位` 写玩家档位和用户预算摘要；大小月卡 68 元档固定写 `大小月卡党（68元档）`，例如 `大小月卡党（68元档） + 6 元方斯礼包 x2`。
- 顶部 `目标` 用汇总口径，不列出每个抽卡目标名；例如 `角色 2 个 / 专武 1 把 / 联动满档 / 200 万房产`。抽卡目标明细只在目标消耗表中逐行展示。
- HTML 模块顺序固定：结论、资源获取进度（仅按日期查询时显示）、可用资源统计、目标消耗、版本活动方案（仅版本资料存在时显示）、综合分析。
- 结论卡片展示版本末环石、版本末方斯、三重钥匙余量和版本活动推荐方案。
- 可用资源统计和目标消耗以表格呈现。
- 按日期查询时，可用资源统计表列顺序为：`资源`、`当前`、`已开放`、`未开放`、`礼包`、`合计`。
- 整版本查询时，可用资源统计表列顺序为：`资源`、`当前`、`v<版本>`、`礼包`、`合计`，例如 `v1.1`。
- 可用资源统计表的 `礼包` 列只写数量或方斯金额，不附加限定归属说明；例如限定骰子礼包写 `2`。限定归属需要说明时，放入表格下方备注或综合分析。
- 模板中的 `EVENT_PLAN_SECTION`：版本资料存在活动支付方案时填入对应模块；没有活动支付方案时填空字符串。
- 目标消耗项目名统一写作：抽卡目标用 `角色 <角色名>`、`武器 <角色名>专武`；非抽卡目标用 `猎人交易所`、版本活动名、`好感度`、`房产载具`。
- 角色与武器目标的消耗列只写资源消耗，不写日期；抽卡道具不足但环石可覆盖时，环石写在消耗列作为补差；环石补差为 0 时省略环石，例如 `60 三重钥匙`。
- 角色与武器目标的备注默认只写卡池时间范围，句式为 `卡池 <开始日期> 至 <结束日期>。`；只有截至卡池结束时间，道具加可用环石仍不足时才写缺口，并补充缺口规模、截止时间和可行补齐路径。
- 版本活动方案表首列写 `方案`，填入支付档位或支付方式；推荐方案在方案文本后加 `（推荐）`。
- 版本资料存在活动支付方案时，按该版本规则输出 3-6 条与约束最相关的方案。
- 综合分析固定 3 个小点，标题依次为 `主要压力`、`推荐方案`、`备选方案`。

### 报告格式稳定约束

生成 HTML 报告时优先保持模板结构和历史报告格式稳定。版式调整仅在用户明确要求时进行；核心表格列、表格形态和小标题段落结构按下列固定格式输出。

方斯用 `万方斯` 简写，按需要保留 1 位小数，例如 `900 万方斯`、`1,372.5 万方斯`。

#### 目标消耗表

固定列顺序：`项目`、`消耗`、`备注`。

| 项目类型 | 项目列 | 消耗列 | 备注列 |
| --- | --- | --- | --- |
| 角色目标 | `角色 <角色名>` | `<限定骰子> 限定骰子`；道具不足但环石可覆盖时追加 `+ <环石> 环石` | 无缺口只写 `卡池 <开始日期> 至 <结束日期>。`；有缺口才写缺口规模、截止时间和补齐路径 |
| 武器目标 | `武器 <角色名>专武` | `<三重钥匙> 三重钥匙`；道具不足但环石可覆盖时追加 `+ <环石> 环石` | 无缺口只写 `卡池 <开始日期> 至 <结束日期>。`；有缺口才写缺口规模、截止时间和补齐路径 |
| 版本活动 | 版本活动名 | 环石成本 + 方斯成本 | 写推荐支付方式和达成结果，不展开计算过程 |
| 猎人交易所 | `猎人交易所` | 方斯成本 | 只写购买内容，例如 `购买环石、三重钥匙和永恒的螺音`，不写其他说明 |
| 好感度 | `好感度` | 方斯成本 | 只写 `<每日成本> 万方斯/天 x <版本天数> 天`，不写其他说明 |
| 房产载具 | `房产载具` | 方斯成本 | 写预留目标 |

补差和缺口分开判断：补差表示抽卡道具不足但环石足够覆盖，是正常消耗，写在消耗列；缺口表示截至卡池结束时，道具加可用环石仍不足，才写入备注和结论。

角色和武器备注保持短句，常规状态只写卡池时间。

#### 版本活动方案表

环石/方斯双支付活动的方案分析表固定列顺序：

| 列 | 写法 |
| --- | --- |
| `方案` | 直接写支付档位组合或支付方式，例如 `15、14 档用环石（推荐）` |
| `联动环石` | 该方案活动环石消耗 |
| `联动方斯` | 该方案活动方斯消耗 |
| `版本末环石` | 扣除所有目标后的版本末环石 |
| `版本末方斯` | 扣除所有目标后的版本末方斯；不足时写缺口 |
| `说明` | 一句短评，例如 `推荐方案，目标和方斯安全垫都稳。` |

推荐行使用 `tr.recommended`。

#### 综合分析

综合分析固定输出为 3 组 `h3 + p`：

```html
<h3>主要压力</h3>
<p>...</p>
<h3>推荐方案</h3>
<p>...</p>
<h3>备选方案</h3>
<p>...</p>
```

生成后自检：

- 目标消耗表的角色和武器备注按常规状态或缺口状态输出。
- 版本活动方案表表头与固定列顺序一致。
- 综合分析中存在 3 个 `h3`，结构为 `h3 + p`。
- HTML 模板占位符已全部替换。

## 分支处理

| 情况 | 处理方式 |
| --- | --- |
| 缺少可获得日期 | 标记为 `待补日期`，不计入已开放/未开放拆分；若整版本汇总保留其数量，在校验状态说明 |
| 缺少数量 | 标记为 `待补数据`，本次不计入资源统计或消耗统计 |
| 卡池时间缺失 | 只做整版本估算，并说明未完成卡池结束时间检查 |
| 版本末资源够但卡池结束前不够 | 报告中按卡池结束时间写出缺口，并给出提前补差或放弃/后移目标的方案 |
| 武器目标未指定对象 | 先按用户输入和档案偏好匹配；仍不明确时写明待确认 |
| 礼包未明确购买 | 作为备选补差说明，不进入礼包列 |
| 体力/活力兑换档位缺数据 | 暂不计入消耗，仅提示数据未补齐 |
| 用户明确不买猎人交易所 | 从版本可用资源和方斯消耗中移除对应项 |

## 表达风格

- 先给结论，再给关键数字和备选方案。
- 报告面向普通玩家，少用内部规划术语。
- 写清余量、缺口、截止时间和未计入项。
- 备注使用自然句，直接说明计算依据和结论。
