# 异环资源规划

根据玩家库存、档案偏好、版本资料和本次目标生成资源规划报告。

## 能力分层

本 skill 按五层维护。新增资料或规则时，先判断所属层级，再写入对应位置。

| 层级 | 内容 | 维护位置 |
| --- | --- | --- |
| 游戏概念层 | 资源、卡池、抽卡、商店、付费、好感度、都市玩法、固定活动资源等长期稳定机制 | `references/fixed-rules.md`、`references/shop-exchange.md`、`references/recurring-events.md` |
| 版本事实层 | 当前版本的版本信息、卡池时间、资源收入、礼包、版本校验和版本特殊活动 | `references/versions/v<版本>/resources.md`、`references/versions/v<版本>/events.md` |
| 用户档案层 | 用户目标优先级、安全垫、付费习惯、武器态度、好感度和都市偏好 | `profiles/<档案名>.md` |
| 分析流程层 | 从输入到报告的标准计算步骤、目标匹配和输出结构 | 本文件 |
| 问题分支层 | 缺数据、目标不明确、卡池时间不足、礼包未明确购买、版本中途输入等处理方式 | 本文件 |

## 资源规划报告

### 资料读取

1. 读取游戏概念层：`references/fixed-rules.md`。
2. 读取版本事实层：优先读取 `references/versions/v<版本>/resources.md`，用于版本信息、卡池时间、日期拆分、资源明细和整版本汇总；存在 `events.md` 时同时读取，用于版本特殊活动和活动方案分析。跨版本预估时，读取当前版本至目标版本中已存在的版本资料；缺少未来版本资料时，使用当前版本收益汇总估算该版本收入，不创建版本文件。
3. 只在涉及商店、体力、活力、材料性价比或方斯补差时读取 `references/shop-exchange.md`。
4. 读取用户档案层：用户明确指定档案时，比如 `我是 <档案名>`、`按 <档案名>`、`使用 <档案名> `、`<档案名> 档案` 时，使用指定档案`profiles/<档案名>.md`；未明确指定档案时，使用默认档案 `profiles/nineo.md`。

用户输入可按 `当前资源`、`氪金预算`、`目标` 分段。

### 氪金预算与收入档位判定

预算分为 `小月卡/大月卡档位` 和 `追加礼包/首充` 两层。分析时先按档案确定小月卡/大月卡档位，再按用户本次输入叠加追加项。

| 输入情况 | 处理方式 |
| --- | --- |
| 档案记录小月卡、大月卡或大小月卡，用户本次只写购买礼包 | 保留档案月卡档位，追加本次明确购买的礼包 |
| 用户本次明确写 `零氪`、`不买月卡`、`只买`、`本次改为`、`不买大月卡` 等覆盖语义 | 按本次输入覆盖对应档案项 |
| 用户本次明确写小月卡或大月卡档位 | 按本次输入更新对应月卡档位；未提及的追加礼包不默认计入 |
| 没有档案付费偏好，也没有本次预算输入 | 按零氪档计算 |

收入档位按合并后的小月卡/大月卡档位选择：无大小月卡时使用 `零氪`；有小月卡、无大月卡时使用 `小月卡党`；有小月卡和大月卡时使用 `大小月卡党`。档案写 `月卡 + 大月卡` 时，`月卡` 视为小月卡。

`氪金预算` 中明确写“计划买 / 会买 / 预算买 / 买”的礼包视为计入礼包列；写“可考虑 / 可能买 / 备选”的礼包只作为备选说明。

### 分析步骤

1. 确认版本、档案、当前库存、合并后的氪金预算和目标。库存必须来自用户输入或档案。
2. 确认资源计算方式：
   - 默认使用 `当前版本预测`，用于判断当前库存到当前版本结束是否够用；查询日期固定为当前日期，不接受指定其他查询日期。
   - 用户明确写 `整版本预估` 时，使用 `整版本预估`，把用户提供的库存视为指定版本期初库存。
   - 目标版本晚于当前版本，或用户写未来版本目标时，使用 `跨版本预估`。例如当前版本为 v1.1，目标写 `1.2 两个角色` 时，计算范围为 v1.1 剩余资源 + v1.2 整版本收入。
   - 跨版本预估缺少未来版本资料时，按当前版本收益汇总估算每个缺失未来版本收入，并在报告中标记为估算。
3. 按用户档案和本次预算输入合并氪金预算，选择收入档位、目标优先级、安全垫和付费习惯；用户本次输入存在覆盖语义时覆盖档案。
4. 匹配目标与卡池。角色目标匹配 `限定棋盘`，武器目标匹配 `弧盘研摹`；抽卡目标按具体角色或武器拆成独立目标，不合并成总数。
5. 计算可用资源：
   - 当前版本预测：`当前库存 + 当前版本查询日后未开放资源 + 计入礼包`。
   - 整版本预估：`期初库存 + 指定版本整版本收入 + 计入礼包`。
   - 跨版本预估：`当前库存 + 当前版本查询日后未开放资源 + 中间完整版本收入 + 目标版本整版本收入 + 计入礼包 * 跨版本计入版本数`。
6. 计算目标消耗：角色、武器、版本活动、好感度、都市消费和计入商店兑换。跨版本预估涉及版本活动时，按活动所属版本的截止时间计算资源和消耗；该截止时间之后的资源不用于该活动。当前版本猎人交易所默认已购买；只有用户明确说未购买或要求补买时，才单独列为补买项。好感度消耗优先读取用户档案 `好感度` 表，按 `价格 x 次数` 计算每日方斯消耗；免费绒绒棉花和免费约会按固定规则计入，表中空白或 `-` 视为未记录。
7. 按卡池结束时间检查角色和武器目标。卡池结束后的资源不用于该卡池目标。
8. 根据环石、方斯、安全垫和目标优先级给出推荐路线与备选路线。
9. 生成 HTML 报告，先给结论，再给关键数字、表格和分支说明。

### 资源计算方式

```text
当前版本预测：
截至查询日期已开放资源 = 可获得日期 <= 查询日期的资源 + 已累计每日资源
查询日期后未开放资源 = 查询日期之后至版本结束仍可获得的资源
可用资源 = 当前库存 + 查询日期后未开放资源 + 计入礼包资源
版本理论总资源 = 截至查询日期已开放资源 + 查询日期后未开放资源

整版本预估：
触发条件 = 用户明确写“整版本预估”
可用资源 = 期初库存 + 指定版本收益汇总中的整版本资源 + 计入礼包资源

跨版本预估：
触发条件 = 目标版本晚于当前版本，或用户写未来版本目标
当前版本剩余资源 = 当前库存 + 当前版本查询日期后未开放资源
中间完整版本收入 = 当前版本之后、目标版本之前各版本的整版本收入
目标版本收入 = 目标版本整版本收入
跨版本计入版本数 = 当前版本、各中间完整版本和目标版本的数量之和
可用资源 = 当前版本剩余资源 + 中间完整版本收入 + 目标版本收入 + 计入礼包资源 * 跨版本计入版本数
缺少未来版本资料时，使用当前版本收益汇总作为该缺失版本的估算收入
```

- 每日资源以具体版本 `resources.md` 的每日资源起止日期为准，按包含首尾日期的自然日计算。
- 周期资源、一次性资源和活动资源按 `可获得日期` 判断是否已经开放。
- 活动奖励默认按活动第一天全部可获得计算；若具体版本有分阶段领取规则，在该版本 `resources.md` 的版本特例中说明。
- 版本收入按分析流程已确定的收入档位读取；追加礼包和首充按分析流程确认的计入项进入礼包列。
- 礼包资源只表示“可购买资源”，不默认计入已获得资源；只有分析流程确认计入时才计入。
- 缺少可获得日期的资源标记为 `待补日期`，不计入已开放/未开放拆分和收益汇总；在待确认项说明。
- 缺少数量的数据标记为 `待补数据`，本次不计入资源统计或消耗统计。

### 卡池与目标匹配

- 角色和武器目标必须结合 `resources.md` 的卡池结束时间检查。版本末资源够用，不代表较早结束的卡池也够用。
- 跨版本预估缺少目标版本卡池资料时，按用户写明的目标数量计算消耗，不做具体卡池结束时间检查；报告备注写明目标版本卡池资料缺失，按版本末可用资源估算。
- `限定棋盘` 对应限定角色目标，消耗 `限定骰子` 和环石补差。
- `弧盘研摹` 对应武器目标，消耗 `三重钥匙` 和环石补差。
- 目标写作 `<角色数>+<武器数>` 时，按命座和专武目标解析：角色目标数量为 `角色数 + 1`，武器目标数量为 `武器数`。例如 `0+1` 表示抽 1 个角色和 1 把武器，`2+1` 表示抽 3 个角色和 1 把武器。
- `上半`、`下半`按同类型卡池时间顺序匹配；例如上半角色匹配第 1 个 `限定棋盘`，上半武器匹配同期第 1 个 `弧盘研摹`，下半角色匹配第 2 个 `限定棋盘`。
- 武器目标指定角色时，匹配与该角色同期的 `弧盘研摹`；报告项目名统一写作 `武器 <角色名>专武`，不展示具体武器名。
- 卡池目标未指定对象，先按用户输入和档案偏好匹配，若仍无法判断，在报告中写明待确认。
- 抽卡目标输出一目标一行：抽几个角色/武器目标就写几行，例如 `角色 安魂曲`、`武器 安魂曲专武`、`角色 卡厄斯`。
- 多个卡池目标按结束时间顺序滚动计算；同一结束时间先列角色、后列武器；已分配给前序目标的抽卡道具和环石不再重复计入后序目标。
- 每个抽卡目标按自身卡池结束时间单独分析资源；只计入该卡池结束时间前可获得的资源和计入的相关礼包，卡池结束后的资源归入后续目标或版本末余量。

### 报告生成流程

- 默认生成 HTML 报告；生成 HTML 报告时使用 `report-template.html` 作为模板，不参考历史报告。
- 报告文件固定保存到仓库根目录 `reports/`。
- 用户明确指定档案时，文件名为 `reports/<档案名>-yyyyMMddHHmm.html`；未明确指定档案时，文件名为 `reports/yyyyMMddHHmm.html`。
- 未明确指定档案但按默认规则读取 `profiles/nineo.md` 时，文件名仍按未指定档案处理，不加 `nineo` 前缀。
- 报告标题默认为 `档案名 <版本> 版本资源分析`；未明确指定档案时写作 `<版本> 版本资源分析`。
- HTML 模块顺序固定：结论、资源统计、目标消耗、版本活动方案、综合分析。
- 模板中的 `EVENT_PLAN_SECTION`：当前版本预测和整版本预估读取报告版本 `events.md` 的 `活动方案分析`；跨版本预估按涉及版本分别读取 `events.md` 的 `活动方案分析`；没有对应模块则填空字符串。
- 版本活动方案按对应版本规则输出 3-6 条与约束最相关的方案；跨版本预估涉及多个版本活动时，分别按所属版本截止时间计算。

### 报告格式稳定约束

生成 HTML 报告时优先保持模板结构和历史报告格式稳定。版式调整仅在用户明确要求时进行；核心表格列、表格形态和小标题段落结构按下列固定格式输出。

方斯默认用 `万方斯` 简写，按需要保留 1 位小数，例如 `900 万方斯`、`1,372.5 万方斯`；表格行标题或列标题已写 `方斯` 时，单元格只写 `<数值> 万`，例如 `900 万`、`1,372.5 万`。

#### 顶部信息与结论

- HTML 顶部只输出标题和元信息。
- 顶部元信息顺序固定为：`生成日期`、`版本`、`目标`、`氪金预算`、`报告类型`。
- `生成日期` 只显示日期，格式固定为 `yyyy-MM-dd`，不显示具体时间。
- `目标` 用汇总写法，例如 `角色 2 个 / 专武 1 把 / 联动满档 / 200 万房产`；抽卡目标明细只在目标消耗表中逐行展示。
- `氪金预算` 写合并后的预算摘要。预算项按小月卡、大月卡档位、追加礼包顺序用 ` + ` 连接；常用写法为 `小月卡`、`大月卡（68元档）`、`6元礼包`，例如 `小月卡 + 大月卡（68元档） + 6元礼包`。同类礼包明确购买多个时在礼包后追加数量，例如 `6元礼包x2`；无付费预算时写 `零氪`。
- `报告类型` 取值：当前版本预测写 `当前版本预测`，整版本预估写 `整版本预估`，跨版本预估写 `跨版本预估`。
- 报告正文涉及当前版本猎人交易所时，使用正向表达 `猎人交易所默认已购买`。只有用户明确说未购买或要求补买时，才写 `若未购买，可另消耗 <方斯> 万方斯购买 <奖励>`。
- 结论卡片展示资源结余、三重钥匙余量和版本活动推荐方案。当前版本预测和整版本预估时，资源结余标签写 `版本末环石`、`版本末方斯`；跨版本预估时写 `跨版本末环石`、`跨版本末方斯`。

#### 资源统计表

- 资源统计和目标消耗以表格呈现。
- 当前版本预测时，资源统计表列顺序为：`资源`、`当前`、`未开放`、`礼包`、`合计`。
- 整版本预估时，资源统计表列顺序为：`资源`、`期初`、`v<版本>`、`礼包`、`合计`，例如 `v1.1`。
- 跨版本预估时，资源统计表列顺序为：`资源`、`当前`、`v<当前版本>未开放`、`v<中间版本>`、`v<目标版本>`、`礼包x版本数`、`合计`；缺少资料并使用当前版本收益估算的版本列标题追加 `（估算）`。
- 资源统计表的 `礼包` 或 `礼包x版本数` 列只写数量或方斯金额，不附加限定归属说明；例如限定骰子礼包写 `2`。

#### 目标消耗表

固定列顺序：`项目`、`消耗`、`备注`。

| 项目类型 | 项目列 | 消耗列 | 备注列 |
| --- | --- | --- | --- |
| 角色目标 | `角色 <角色名>` | `<限定骰子> 限定骰子`；道具不足但环石可覆盖时追加 `+ <环石> 环石` | 无缺口只写 `卡池 <开始日期> - <结束日期>。`；有缺口才写缺口规模、截止时间和补齐路径 |
| 武器目标 | `武器 <角色名>专武` | `<三重钥匙> 三重钥匙`；道具不足但环石可覆盖时追加 `+ <环石> 环石` | 无缺口只写 `卡池 <开始日期> - <结束日期>。`；有缺口才写缺口规模、截止时间和补齐路径 |
| 版本活动 | 版本活动名 | 环石成本 + 方斯成本 | 写推荐支付方式和达成结果，不展开计算过程 |
| 猎人交易所 | `猎人交易所` | 方斯成本 | 只写购买内容，例如 `购买环石、三重钥匙和永恒的螺音`，不写其他说明 |
| 好感度 | `好感度` | 方斯成本 | 只写 `<每日成本> 万方斯/天 x <版本剩余天数> 天`，不写其他说明 |
| 房产载具 | `房产载具` | 方斯成本 | 写预留目标 |

- 角色与武器目标的消耗列只写资源消耗；抽卡道具不足但环石可覆盖时，环石写在消耗列作为补差；环石补差为 0 时省略环石，例如 `60 三重钥匙`。
- 角色与武器目标的备注默认只写卡池时间范围；只有截至卡池结束时间，道具加可用环石仍不足时才写缺口，并补充缺口规模和可行补齐路径。
- 补差和缺口分开判断：补差表示抽卡道具不足但环石足够覆盖，是正常消耗，写在消耗列；缺口表示截至卡池结束时，道具加可用环石仍不足，才写入备注和结论。

#### 版本活动方案

- 只有对应版本 `events.md` 存在 `活动方案分析` 时，才生成版本活动方案模块；没有则不输出该版本活动方案。
- 版本活动方案的内容、推荐规则和展示方式以对应版本 `events.md` 为准。
- 跨版本预估涉及版本活动时，每个活动按所属版本截止时间计算；若活动资料有独立截止时间，使用活动截止时间。
- 版本活动方案表中的 `版本末环石`、`版本末方斯` 按该活动所属版本的版本末资源计算；跨版本预估时，不使用跨版本末资源填入当前版本活动方案。
- 推荐方案行使用 `tr.recommended`。

#### 综合分析

综合分析固定输出为 3 组 `h3 + p`：

```html
<h3>主要压力</h3>
<p>...</p>
<h3>推荐方案</h3>
<p>...</p>
<h3>备选方案</h3>
<p>...</p>
```

### 生成后自检

- 目标消耗表的角色和武器备注按常规状态或缺口状态输出。
- 版本活动方案表头与当前版本 `events.md` 的 `活动方案分析` 一致。
- 综合分析中存在 3 个 `h3`，结构为 `h3 + p`。
- HTML 模板占位符已全部替换。

### 特殊情况处理

| 情况 | 处理方式 |
| --- | --- |
| 版本末资源够但卡池结束前不够 | 按卡池结束时间写缺口，并给出提前补差或放弃方案 |
| 用户明确不买猎人交易所 | 从版本可用资源和方斯消耗中移除对应项 |

## 游戏资料维护

新增资料时按以下顺序处理：

1. 判定层级：长期机制进游戏概念层、版本日历进版本事实层、玩家偏好进用户档案层。
2. 确保 `references/versions/v<版本>/` 存在；`resources.md` 为版本主文件，缺少时复制 `references/version-resources-template.md`；只有存在版本特殊活动或活动方案分析时，才维护 `events.md`，缺少时复制 `references/version-events-template.md`。
3. 从版本日历整理 `卡池信息`：`卡池名称` 填卡池横幅标题，`卡池内容` 填图中明确展示的角色名或武器名。
4. 整理每日、周期、活动和一次性资源，写清资源类型、数量、可获得日期和备注；活动资源先按 `references/recurring-events.md` 匹配，命中固定活动资源时按当前版本日历日期套用模板，命中无核心资源活动时不写入 `resources.md`，未命中且缺少数量时写入待确认项；常规礼包按 `references/fixed-rules.md` 自动生成，只有特殊礼包写入 `resources.md`，包括礼包内容、价格、限购、上架规则变化或版本专属礼包；版本特殊活动、专属活动档位和活动方案分析写入 `events.md`。
5. `收益汇总` 由本文件的每日、周期、活动和一次性资源明细按档位计算生成，不保留外部汇总表原值；缺日期、缺数量或不计入汇总的项目写入待确认项。

## 用户档案维护

只有用户明确发出 `建立档案{name}`、`建档{name}` 指令时，才建立新档案。建档时复制 `profiles/profile-template.md`，保存为 `profiles/<name>.md`。

建档流程：

1. 从指令中提取 `{name}`。
2. 复制 `profiles/profile-template.md`，保存为 `profiles/<name>.md`。
3. 将用户本次提供的长期偏好写入新档案；未提供的项目保留模板占位。

未出现 `给{name}建档` 指令时，不创建新档案。当前库存、当前版本目标、临时预算、单次礼包购买、某个版本临时跳过或追加角色、一次性买房或载具目标，都保留为本次输入或报告备注。
