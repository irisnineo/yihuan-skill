# 异环资源规划

根据玩家库存、档案偏好、版本资料和本次目标生成资源规划报告。

## 能力分层

本技能按五层维护。新增资料或规则时，先判断所属层级，再写入对应位置。

| 层级 | 内容 | 维护位置 |
| --- | --- | --- |
| 游戏概念层 | 资源、卡池、抽卡、商店、付费、好感度、都市玩法等长期稳定机制 | `references/fixed-rules.md`、`references/shop-exchange.md` |
| 版本事实层 | 当前版本的卡池时间、资源收入、活动、礼包、版本校验和版本专属活动档位 | `references/versions/v<版本>/main.md`、`references/versions/v<版本>/progress.md` |
| 玩家档案层 | 玩家目标优先级、安全垫、付费习惯、武器态度、好感度和都市偏好 | `profiles/<档案名>.md` |
| 分析流程层 | 从输入到报告的标准计算步骤、目标匹配和输出结构 | 本文件 |
| 问题分支层 | 缺数据、目标不明确、卡池时间不足、礼包未明确购买、版本中途输入等处理方式 | 本文件的“分支处理” |

## 资料读取

1. 读取游戏概念层：`references/fixed-rules.md`。
2. 读取版本事实层：`references/versions/v<版本>/main.md`；存在 `progress.md` 时同时读取，用于卡池时间、日期拆分和整版本汇总。
3. 只在涉及商店、体力、活力、材料性价比或方斯补差时读取 `references/shop-exchange.md`。
4. 读取玩家档案层：用户指定 `profiles/<档案名>.md` 时使用指定档案；未指定时使用 `profiles/nineo.md`。用户本次输入优先于档案。
5. 生成 HTML 报告时使用 `templates/resource-report.html`。

用户输入可按 `当前资源`、`氪金预算`、`目标` 分段。

### 氪金预算与收入档位判定

预算分为 `基础订阅/月卡档位` 和 `追加礼包/首充` 两层。分析时先按档案确定基础订阅/月卡档位，再按用户本次输入叠加追加项。

| 输入情况 | 处理方式 |
| --- | --- |
| 档案记录小月卡、大月卡或大小月卡，用户本次只写购买礼包 | 保留档案月卡档位，追加本次明确购买的礼包 |
| 用户本次明确写 `零氪`、`不买月卡`、`只买`、`本次改为`、`不买大月卡` 等覆盖语义 | 按本次输入覆盖对应档案项 |
| 用户本次明确写小月卡或大月卡档位 | 按本次输入更新对应月卡档位；未提及的追加礼包不默认计入 |
| 没有档案付费偏好，也没有本次预算输入 | 按零氪档计算 |

收入档位按合并后的基础订阅/月卡档位选择：无小月卡和大月卡时使用 `零氪`；有小月卡、无大月卡时使用 `小月卡党`；有小月卡和大月卡时使用 `大小月卡党`。档案写 `月卡 + 大月卡` 时，`月卡` 视为小月卡。

`氪金预算` 中明确写“计划买 / 会买 / 预算买 / 买”的礼包视为计入礼包列；写“可考虑 / 可能买 / 备选”的礼包只作为备选说明。用户写 `买 6r 方斯礼包` 或 `买 6 元方斯礼包` 且未指定数量时，按当版本 2 个角色单抽方斯礼包计入；报告展示统一写作 `购买 6 元方斯礼包x2`。

## 玩家档案维护

只有用户明确发出 `给{name}建档` 指令时，才建立新玩家档案。建档时复制 `profiles/profile-template.md`，保存为 `profiles/<name>.md`。

建档流程：

1. 从指令中提取 `{name}`，生成文件 `profiles/<name>.md`。
2. 复制 `profiles/profile-template.md` 的结构。
3. 将用户本次提供的长期偏好写入新档案；未提供的项目保留模板占位。

未出现 `给{name}建档` 指令时，不创建新档案。当前库存、当前版本目标、临时预算、单次礼包购买、某个版本临时跳过或追加角色、一次性买房或载具目标，都保留为本次输入或报告备注。

## 资料维护流程

新增资料时按以下顺序处理：

1. 判定层级：长期机制进游戏概念层、版本日历进版本事实层、玩家偏好进玩家档案层。
2. 确保 `references/versions/v<版本>/` 存在；缺少 `progress.md` 时复制 `references/version-progress-template.md`。
3. 从版本日历整理 `卡池信息`：`卡池名称` 填卡池横幅标题，`卡池内容` 填图中明确展示的角色名或武器名。
4. 整理每日、周期、活动、一次性和礼包资源，写清资源类型、数量、可获得日期、档位和领取方式；礼包资源同时记录 `礼包类型` 和游戏内显示名。
5. 对齐整版本汇总；存在差异时写入校验状态或待确认项。

## 标准分析流程

1. 确认版本、档案、当前库存、合并后的氪金预算和目标。库存必须来自用户输入或档案。
2. 确认资源计算方式：
   - 默认使用 `进行中拆分`，查询日期固定为当前日期；不接受指定其他查询日期。
   - 用户写 `整版本预估`、`版本预估`、`版本前` 或 `还没开始玩` 时，使用整版本汇总。
   - 用户写“版本 X + 当前资源”时，默认理解为规划版本 X 的期初库存；只有明确写“当前剩余资源 / 已领取后的余量 / 版本进行中余量”时，才不重复叠加已开放资源。
3. 按玩家档案和本次预算输入合并氪金预算，选择收入档位、目标优先级、安全垫和付费习惯；用户本次输入存在覆盖语义时覆盖档案。
4. 匹配目标与卡池。角色目标匹配 `限定棋盘`，武器目标匹配 `弧盘研摹`；抽卡目标按具体角色或武器拆成独立目标，不合并成总数。
5. 计算可用资源：
   - 进行中拆分：`当前库存 + 已确定收入档位的截至查询日已开放资源 + 查询日后未开放资源 + 计入礼包`。
   - 整版本预估：`当前库存 + 已确定收入档位的整版本收入 + 计入礼包`。
6. 计算目标消耗：角色、武器、活动、好感度、都市消费和计入商店兑换。版本进行中，默认猎人交易所已购买；只有用户明确说未购买或要求补买时，才单独列为补买项。好感度消耗优先读取玩家档案 `好感度` 表，按 `价格 x 次数` 计算每日方斯消耗；免费云朵道具和免费约会按固定规则计入，表中空白或 `-` 视为未记录。
7. 按卡池结束时间检查角色和武器目标。卡池结束后的资源不用于该卡池目标。
8. 根据环石、方斯、安全垫和目标优先级给出推荐路线与备选路线。
9. 生成 HTML 报告，先给结论，再给关键数字、表格和分支说明。

## 资源计算方式

```text
进行中拆分：
截至查询日期已开放资源 = 可获得日期 <= 查询日期的资源 + 已累计每日资源
查询日期后未开放资源 = 查询日期之后至版本结束仍可获得的资源
可用资源 = 当前库存 + 截至查询日期已开放资源 + 查询日期后未开放资源 + 计入礼包资源
当前库存为当前剩余资源/已领取后的余量/版本进行中余量时，可用资源 = 当前库存 + 查询日期后未开放资源 + 计入礼包资源，已开放资源只用于进度展示
版本理论总资源 = 截至查询日期已开放资源 + 查询日期后未开放资源

整版本预估：
可用资源 = 当前库存 + 汇总校验中的整版本资源 + 计入礼包资源
```

- 每日资源以具体版本 `progress.md` 的每日资源起止日期为准，按包含首尾日期的自然日计算。
- 周期资源、一次性资源和活动资源按 `可获得日期` 判断是否已经开放。
- 活动奖励默认按活动第一天全部可获得计算；若具体版本有分阶段领取规则，在该版本 `progress.md` 的版本特例中说明。
- 版本收入按分析流程已确定的收入档位读取；追加礼包和首充按分析流程确认的计入项进入礼包列。
- 礼包资源只表示“可购买资源”，不默认计入已获得资源；只有分析流程确认计入时才计入。
- 缺少可获得日期的资源标记为 `待补日期`，不计入已开放/未开放拆分；如整版本汇总保留其数量，在校验状态说明。
- 缺少数量的数据标记为 `待补数据`，本次不计入资源统计或消耗统计。

## 卡池与目标匹配

- 如果版本 `progress.md` 存在 `卡池信息`，角色和武器目标必须同时做卡池结束时间检查。版本末资源够用，不代表较早结束的卡池也够用。
- `限定棋盘` 对应限定角色目标，消耗 `限定骰子` 和环石补差。
- `弧盘研摹` 对应武器目标，消耗 `三重钥匙` 和环石补差。
- 目标写“角色全拿”或“角色 X 个”且未指定具体角色时，按 `限定棋盘` 的时间顺序匹配版本内限定角色，角色名取 `卡池内容`。
- `上半`、`下半`按同类型卡池时间顺序匹配；例如上半角色匹配第 1 个 `限定棋盘`，上半武器匹配同期第 1 个 `弧盘研摹`，下半角色匹配第 2 个 `限定棋盘`。
- 武器目标指定角色时，匹配与该角色同期的 `弧盘研摹`；报告项目名统一写作 `武器 <角色名>专武`，不展示具体武器名。
- 角色和武器目标无法判断时，在报告中写明待确认。
- 抽卡目标输出一目标一行：抽几个角色/武器目标就写几行，例如 `角色 安魂曲`、`武器 安魂曲专武`、`角色 卡厄斯`。
- 多个卡池目标按结束时间顺序滚动计算；同一结束时间先列角色、后列武器；已分配给前序目标的抽卡道具和环石不再重复计入后序目标。
- 每个抽卡目标按自身卡池结束时间单独分析资源；只计入该卡池结束时间前可获得的资源和计入的相关礼包，卡池结束后的资源归入后续目标或版本末余量。

## 报告生成流程

- 默认生成 HTML 报告；用户明确指定档案时，文件名为 `reports/<档案名>-yyyyMMddHHmm.html`；未明确指定档案时，文件名为 `reports/yyyyMMddHHmm.html`。
- 未明确指定档案但按默认规则读取 `profiles/nineo.md` 时，文件名仍按未指定档案处理，不加 `nineo` 前缀。
- 报告标题默认为 `档案名 <版本> 版本资源分析`；未明确指定档案时写作 `<版本> 版本资源分析`。
- HTML 模块顺序固定：结论、资源获取进度（仅进行中拆分显示）、可用资源统计、目标消耗、版本活动方案（仅版本资料存在时显示）、综合分析。
- 模板中的 `EVENT_PLAN_SECTION`：版本资料存在活动支付方案时填入对应模块；没有活动支付方案时填空字符串。
- 版本资料存在活动支付方案时，按该版本规则输出 3-6 条与约束最相关的方案。

## 报告格式稳定约束

生成 HTML 报告时优先保持模板结构和历史报告格式稳定。版式调整仅在用户明确要求时进行；核心表格列、表格形态和小标题段落结构按下列固定格式输出。

方斯默认用 `万方斯` 简写，按需要保留 1 位小数，例如 `900 万方斯`、`1,372.5 万方斯`；表格行标题或列标题已写 `方斯` 时，单元格只写 `<数值> 万`，例如 `900 万`、`1,372.5 万`。

### 顶部信息与结论

- 报告副标题用于介绍本次计算现状，按报告类型分别输出。
- 进行中拆分副标题固定顺序为：`当前库存`、`抽取情况`、`特殊项`。
- 进行中拆分副标题写法：`当前库存：环石 <数量>、限定骰子 <数量>、三重钥匙 <数量>、方斯 <数量> 万；抽取情况：<抽取进度>；<特殊项>。`
- `抽取情况` 只写用户本次明确提供的角色、武器、联动或活动抽取进度；未提供时省略整段。
- 整版本预估副标题开头固定写 `版本开始前库存：...`，写法为：`版本开始前库存：环石 <数量>、限定骰子 <数量>、三重钥匙 <数量>、方斯 <数量> 万；<特殊项>。`
- `特殊项` 写用户本次明确输入，或用户未提及但会影响后续计算的内容；只写实际影响本次计算的项目。没有特殊项时省略该段。
- 顶部元信息顺序固定为：`生成日期`、`版本`、`氪金预算`、`目标`、`报告类型`。`报告类型` 取值：整版本汇总写 `整版本预估`，进行中拆分写 `版本进行中`。进行中拆分不单独展示查询日期，生成日期即查询日期。
- `氪金预算` 写合并后的预算摘要。预算项按小月卡、大月卡档位、追加礼包顺序用 ` + ` 连接；常用写法为 `小月卡`、`大月卡（68元档）`、`6元礼包`，例如 `小月卡 + 大月卡（68元档） + 6元礼包`。同类礼包明确购买多个时在礼包后追加数量，例如 `6元礼包x2`；无付费预算时写 `零氪`。
- 顶部 `目标` 用汇总写法，例如 `角色 2 个 / 专武 1 把 / 联动满档 / 200 万房产`；抽卡目标明细只在目标消耗表中逐行展示。
- 报告正文涉及版本进行中的猎人交易所时，使用正向表达 `猎人交易所默认已购买`。只有用户明确说未购买或要求补买时，才写 `若未购买，可另消耗 <方斯> 万方斯购买 <奖励>`。
- 结论卡片展示版本末环石、版本末方斯、三重钥匙余量和版本活动推荐方案。

### 资源获取进度表

- 资源获取进度、可用资源统计和目标消耗以表格呈现。
- 资源获取进度表固定两列：`项目`、`内容`。行顺序为：`已开放资源`、`未开放资源`、`待确定资源`；资源内容只汇总环石、限定骰子、三重钥匙和方斯，格式如 `3,930 环石、10 限定骰子、45 三重钥匙、279.3 万方斯。`，不写其他说明。

### 可用资源统计表

- 进行中拆分时，可用资源统计表列顺序为：`资源`、`当前`、`未开放`、`礼包`、`合计`。
- 整版本查询时，可用资源统计表列顺序为：`资源`、`当前`、`v<版本>`、`礼包`、`合计`，例如 `v1.1`。
- 可用资源统计表的 `礼包` 列只写数量或方斯金额，不附加限定归属说明；例如限定骰子礼包写 `2`。

### 目标消耗表

固定列顺序：`项目`、`消耗`、`备注`。

| 项目类型 | 项目列 | 消耗列 | 备注列 |
| --- | --- | --- | --- |
| 角色目标 | `角色 <角色名>` | `<限定骰子> 限定骰子`；道具不足但环石可覆盖时追加 `+ <环石> 环石` | 无缺口只写 `卡池 <开始日期> - <结束日期>。`；有缺口才写缺口规模、截止时间和补齐路径 |
| 武器目标 | `武器 <角色名>专武` | `<三重钥匙> 三重钥匙`；道具不足但环石可覆盖时追加 `+ <环石> 环石` | 无缺口只写 `卡池 <开始日期> - <结束日期>。`；有缺口才写缺口规模、截止时间和补齐路径 |
| 版本活动 | 版本活动名 | 环石成本 + 方斯成本 | 写推荐支付方式和达成结果，不展开计算过程 |
| 猎人交易所 | `猎人交易所` | 方斯成本 | 只写购买内容，例如 `购买环石、三重钥匙和永恒的螺音`，不写其他说明 |
| 好感度 | `好感度` | 方斯成本 | 只写 `<每日成本> 万方斯/天 x <版本天数> 天`，不写其他说明 |
| 房产载具 | `房产载具` | 方斯成本 | 写预留目标 |

- 角色与武器目标的消耗列只写资源消耗；抽卡道具不足但环石可覆盖时，环石写在消耗列作为补差；环石补差为 0 时省略环石，例如 `60 三重钥匙`。
- 角色与武器目标的备注默认只写卡池时间范围；只有截至卡池结束时间，道具加可用环石仍不足时才写缺口，并补充缺口规模和可行补齐路径。

补差和缺口分开判断：补差表示抽卡道具不足但环石足够覆盖，是正常消耗，写在消耗列；缺口表示截至卡池结束时，道具加可用环石仍不足，才写入备注和结论。

### 版本活动方案表

环石/方斯双支付活动的方案分析表固定列顺序：

| 列 | 写法 |
| --- | --- |
| `方案` | 直接写支付档位组合或支付方式，例如 `15、14 档用环石（推荐）` |
| `联动环石` | 该方案活动环石消耗 |
| `联动方斯` | 该方案活动方斯消耗 |
| `版本末环石` | 扣除所有目标后的版本末环石 |
| `版本末方斯` | 扣除所有目标后的版本末方斯；不足时写缺口 |
| `说明` | 一句短评，例如 `推荐方案，目标和方斯安全垫都稳。` |

- 版本活动方案表首列写 `方案`，填入支付档位或支付方式；推荐方案在方案文本后加 `（推荐）`。

推荐行使用 `tr.recommended`。

## 生成后自检

- 目标消耗表的角色和武器备注按常规状态或缺口状态输出。
- 版本活动方案表表头与固定列顺序一致。
- 综合分析中存在 3 个 `h3`，结构为 `h3 + p`。
- HTML 模板占位符已全部替换。

## 分支处理

| 情况 | 处理方式 |
| --- | --- |
| 缺少可获得日期 | 标记为 `待补日期`，不计入已开放/未开放拆分；若整版本汇总保留其数量，在校验状态说明 |
| 缺少数量 | 标记为 `待补数据`，本次不计入资源统计或消耗统计 |
| 卡池时间缺失 | 只做整版本估算，并说明未完成卡池结束时间检查 |
| 版本末资源够但卡池结束前不够 | 报告中按卡池结束时间写出缺口，并给出提前补差或放弃/后移目标的方案 |
| 武器目标未指定对象 | 先按用户输入和档案偏好匹配；仍不明确时写明待确认 |
| 礼包未明确购买 | 作为备选补差说明，不进入礼包列 |
| 体力/活力兑换档位缺数据 | 暂不计入消耗，仅提示数据未补齐 |
| 用户明确不买猎人交易所 | 从版本可用资源和方斯消耗中移除对应项 |
