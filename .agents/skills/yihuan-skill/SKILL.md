# 异环资源规划

根据玩家库存、档案偏好、版本资料和本次目标生成资源规划报告。

## 资料读取

- 稳定规则：读取 `references/fixed-rules.md`。
- 版本资料：读取 `references/versions/v<版本>/main.md`；存在 `progress.md` 时同时读取，用于日期拆分和整版本汇总。
- 商店、体力、活力、材料性价比或方斯补差：只在相关问题中读取 `references/shop-exchange.md`。
- 玩家偏好：读取用户指定的 `profiles/<档案名>.md`；未指定时使用 `profiles/nineo.md`。用户本次输入优先于档案。
- HTML 报告：使用 `templates/resource-report.html`。

## 资料边界

- 规划依据只使用 `references/`、`profiles/` 和用户本次输入。
- `sources/` 是原始资料和备份，只读处理；不要修改、重命名、移动，也不要自动整理进 `references/`。
- 只有用户明确要求“从 sources 生成/更新 reference”时，才根据 `sources/` 整理资料。
- `references/` 只记录可复用规则和数值；来源、整理经过、当次用户指令等过程信息不写入 reference。
- 如发现 `sources/` 有可用新资料，只提示用户确认后再转入 `references/`。

## 日期资源维护

用户明确要求更新某版本日期资源时：

1. 将版本统一映射为 `v<版本>`，例如 `1.1` -> `v1.1`。
2. 确保 `references/versions/v<版本>/` 存在；缺少 `progress.md` 时复制 `references/version-progress-template.md`。
3. 整理每日、周期、一次性、活动和礼包资源，写清资源类型、数量、可获得日期、档位和领取口径。
4. 缺日期或数量的数据标为 `待补日期` 或 `待补数据`，不要计入已开放/未开放拆分。
5. 对齐整版本汇总；对不齐时在校验表说明差异，不静默改数。

## 规划流程

1. 确认版本、档案和当前库存；库存必须来自用户输入或档案，不从版本资料推断。
2. 确认资源口径：
   - 默认使用 `进行中拆分`，查询日期为当前日期；用户指定日期时用用户日期。
   - 用户写 `整版本预估`、`版本预估`、`版本开始前` 或 `从零开始规划` 时，使用整版本汇总，不按日期拆分。
   - 用户写“版本 X + 当前资源”时，默认理解为规划版本 X 的期初库存；只有明确写“当前剩余资源 / 已领取后的余量 / 版本进行中余量”时，才不重复叠加已开放资源。
3. 读取相关规则和版本文件，按用户或档案选择收入档位。
4. 统一术语：限定角色池道具写 `限定骰子`，常驻池道具写 `常驻骰子`；除非用户明确允许，常驻资源不计入限定角色或武器目标。
5. 计算可用资源统计：
   - 进行中拆分：`当前库存 + 截至查询日已开放资源 + 查询日后未开放资源 + 已确认礼包`。
   - 整版本预估：`当前库存 + 整版本收入 + 已确认礼包`。
6. 计算目标消耗：角色、武器、活动、好感度、都市消费和已确认商店兑换。体力/活力兑换档位仍为待补数据时，不计入消耗。
7. 目标消耗表按玩家档案目标优先级排序；与上级目标直接相关的支撑开销紧跟该目标展示，例如猎人交易所跟随武器目标。
8. 活动支付方案按版本资料计算；根据档案安全垫、环石余量和方斯余量给出推荐与备选方案。

## 报告输出

- 默认生成 HTML 到 `reports/name-yyyyMMdd.html`，标题为 `档案名 v版本 资源分析`。
- 默认输出 HTML 报告；只有用户明确要求不生成文件时，才在对话中给精简文本结论。
- 整版本预估报告副标题写作 `按版本开始前状态进行整版本预估：...`，避免使用“口径计算”这类说法。
- 结论卡片顺序优先写：版本末环石、版本末方斯、三重钥匙余量、推荐联动环石。
- HTML 模块顺序固定：结论、资源获取进度（仅按日期查询时显示）、可用资源统计、目标消耗、联动方案分析、综合分析。
- `资源获取进度` 只在使用 `进行中拆分` 时显示，展示查询日期、版本进度、已开放资源和未开放资源；整版本预估时不显示该模块。
- 模板中的 `PROGRESS_SECTION`：进行中拆分时填入 `<section><h2>资源获取进度</h2>...</section>`；整版本预估时填空字符串。
- 模板中的 `RESOURCE_HEADER_CELLS` 按查询方式填充；表头列数必须与 `RESOURCE_ROWS` 保持一致。
- 可用资源统计和目标消耗只输出表格，不追加描述段。
- 按日期查询时，可用资源统计表列顺序为：`资源`、`当前`、`已开放`、`未开放`、`礼包`、`合计`。
- 整版本查询时，可用资源统计表列顺序为：`资源`、`当前`、`v<版本>`、`礼包`、`合计`，例如 `v1.1`；不要显示 `已开放`、`未开放` 或 `未拆分`。
- 目标消耗项目名统一写作：`角色 X 个`、`武器 X 把`、`猎人交易所`、`保时捷联动`、`好感度`、`买房`。
- 角色与武器目标的消耗列写成 `已用抽卡道具 + 环石补差`；环石补差为 0 时省略环石，例如 `60 三重钥匙`，不要写 `+ 0 环石`。
- 猎人交易所按每版本常规购买计入；具体项目、数量和成本读取 `references/shop-exchange.md`。
- 目标消耗表中的 `保时捷联动` 行使用 `recommended` 整行浅黄高亮。
- 目标消耗备注面向普通玩家，直接写计算依据和结论；不要用 `口径：`、`用途：`、`方案：` 这类标签式开头。
- 方斯数值统一写作 `XXX 万`，数字和 `万` 中间保留一个空格；非整万保留一位小数，例如 `178.5 万方斯`。
- 联动方案只列连续后段方案：从某一档开始到最后一档用环石，前段用方斯，输出 3-6 条与约束最相关的方案。
- 联动方案分析中的推荐方案行使用 `recommended` 整行浅黄高亮，并在档位文本中标注 `（推荐）`。
- 综合分析固定 3 个小点，标题依次为 `主要压力`、`推荐方案`、`备选方案`；每点用一到两句话说明资源压力、推荐路线、切换条件和关键风险。
- 不单独硬编码玩家优先级；优先级、礼包接受度、安全垫等偏好都从玩家档案和本次输入读取。
- 未确认数据写明 `待补数据，本次未计入`，不要自行补全。

## 表达要求

- 先给结论，再给关键数字和备选方案。
- 避免“可能还行”“压力不大”这类模糊话，写清余量、缺口或未计入项。
- 礼包建议合并进备选方案或补差说明，不单独堆成一段。
- 报告面向普通游戏玩家，少用内部规划术语，说明要直观。
