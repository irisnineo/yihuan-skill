# 版本日期资源进度模板

复制本文件到 `references/versions/v<版本>/progress.md` 后维护具体版本数据，例如 `references/versions/v1.1/progress.md`。版本目录名必须带 `v` 前缀，例如 `v1.1`、`v1.2`。

本文件只定义日期拆分口径。不要把 `sources/` 中的日历、礼包图或全勤资源汇总自动写入本文件；只有用户明确要求“根据 sources 更新 progress reference”时，才整理到具体版本文件。

## 计算规则

资源口径默认使用 `进行中拆分`：查询日期默认使用当前日期，用户明确指定日期时才覆盖。
用户明确要求 `整版本预估`、`版本预估`、`还没开始玩` 或 `从零开始规划` 时，使用 `整版本预估`：不按查询日期拆分，直接使用汇总校验中的整个版本资源。

```text
进行中拆分：
截至查询日已开放资源 = 可获得日期 <= 查询日期的资源 + 已累计每日资源
查询日后未开放资源 = 查询日期之后至版本结束仍可获得的资源
版本理论总资源 = 截至查询日已开放资源 + 查询日后未开放资源

整版本预估：
版本可用资源 = 当前库存 + 汇总校验中的整个版本资源 + 已确认购买的礼包资源
```

- 每日资源按包含首尾日期的自然日计算，并限制在版本开始与版本结束之间。
- 周期资源和一次性资源按 `可获得日期` 判断是否已经开放。
- 活动奖励默认按活动第一天全部可获得计算；`可获得日期` 填活动开始日期，减轻版本中途分析压力。
- 礼包资源只表示“可购买资源”，不默认计入已获得资源；只有用户确认购买或预算规则明确时才计入。
- 如果某项资源缺少可获得日期，标记为 `待补日期`，不要计入已开放/未开放拆分。
